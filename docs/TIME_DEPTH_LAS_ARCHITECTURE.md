# Архитектура LAS с глубинными и временными индексами

## 1. Цель

Data Core должен одинаково безопасно работать с:

- LAS, индексированным по глубине;
- LAS, индексированным по относительному или абсолютному времени;
- LAS с несколькими индексами, например TIME + MD + TVD;
- производственными LAS с частично нестандартным заголовком или представлением времени.

Первая колонка не считается глубиной автоматически. Ни одна индексная колонка не
удаляется при выборе другой оси отображения.

## 2. Доменная модель

```python
class IndexType(StrEnum):
    DEPTH = "depth"
    TIME = "time"
    DATETIME = "datetime"
    GENERIC = "generic"
    UNKNOWN = "unknown"

class IndexRole(StrEnum):
    MD = "md"
    TVD = "tvd"
    TVDSS = "tvdss"
    RELATIVE_TIME = "relative_time"
    ABSOLUTE_TIME = "absolute_time"
    GENERIC = "generic"

@dataclass(frozen=True, slots=True)
class DatasetIndex:
    curve_id: str
    index_type: IndexType
    role: IndexRole
    unit: str | None
    confidence: float
    datetime_format: str | None
    timezone: str | None
    start_value: object | None
    stop_value: object | None
    average_step: float | None
    is_monotonic: bool
```

`Dataset` последовательно расширяется полями:

- `indexes: dict[str, DatasetIndex]`;
- `active_index_id: str`;
- `source_format` и `source_format_version`;
- `import_mode`;
- `import_issues`;
- `source_fingerprint` и provenance преобразований.

Числовые кривые остаются NumPy-массивами. Временные значения после нормализации хранятся
как `datetime64[ns]` в UTC либо как числовая относительная шкала с единицей. Строковое
представление источника сохраняется в source snapshot или provenance, а не смешивается с
нормализованной колонкой.

Текущее поле `Dataset.depth` временно остаётся compatibility view активного глубинного
индекса. Оно не должно становиться контейнером времени. Удаление этого compatibility API
возможно только через отдельную миграцию после перевода планшета и расчётов на
`active_index`.

## 3. Результат распознавания

`IndexDetectionService` анализирует каждую колонку и возвращает ранжированные кандидаты,
а не единственный необъяснимый ответ:

```python
@dataclass(frozen=True, slots=True)
class IndexCandidate:
    curve_id: str
    index_type: IndexType
    role: IndexRole
    confidence: float
    evidence: tuple[str, ...]
    warnings: tuple[str, ...]
```

Признаки: мнемоника, единица, описание, тип значений, диапазон, монотонность, шаг,
уникальность, возможность разбора даты, диапазон Unix timestamp и LAS-метаданные.

Пороговая политика:

- высокая уверенность — индекс выбирается автоматически и решение журналируется;
- несколько близких кандидатов — пользователь выбирает основной индекс;
- низкая уверенность — импорт не угадывает молча и переходит в ручную настройку.

Словари мнемоник и единиц являются данными/правилами сервиса и расширяются без изменения
парсера. Для DEPTH учитываются DEPT, DEPTH, MD, TVD, TVDSS, HOLEDEPTH и BITDEPTH; для
времени — TIME, DATETIME, TIMESTAMP, ETIME, EPOCH, UNIXTIME, RECORD_TIME и LOGTIME.

## 4. Конвейер импорта

```text
Source bytes
  -> SourceProbe (кодировка, версия, секции, разделитель)
  -> LasHeaderParser
  -> LasDataParser
  -> RawDataset (без потери колонок и порядка)
  -> IndexDetectionService
  -> TimeNormalizationService / DepthNormalizationService
  -> SemanticChannelDictionary + UomDictionary
  -> DataQualityService
  -> Import Review (read-only model; interactive acceptance next)
  -> Dataset + ImportReport + provenance
```

Стандартный путь может использовать `lasio`. Перед ним выполняется `SourceProbe`, после
него — независимая проверка результата. Резервный tolerant parser допускается только для
совместимого/ручного режима и обязан формировать журнал исправлений.

### Строгий режим

Проверяет структуру, обязательные секции, число колонок и строк. Несоответствие является
ошибкой; автоматическое исправление запрещено.

### Совместимый режим (по умолчанию)

Может нормализовать пробелы, регистр секций, десятичные запятые, пустые описания,
повторяющиеся мнемоники и несовпадения колонок. Каждое действие добавляется в
`ImportReport`; исходные байты не изменяются.

### Ручной режим

Позволяет задать кодировку, начало данных, разделитель, NULL, порядок и тип колонок,
глубину, время, единицы, формат даты и часовой пояс. Предпросмотр выполняется до создания
dataset.

## 5. Нормализация времени

Поддерживаемые входы:

- относительные секунды, миллисекунды, минуты и часы;
- Unix timestamp s/ms/us/ns;
- ISO 8601 и явно выбранные строковые форматы;
- DATE + TIME;
- YEAR/MONTH/DAY/HOUR/MINUTE/SECOND.

Абсолютное время нормализуется в UTC. Исходный часовой пояс и выбранная зона отображения
хранятся отдельно. Если зона отсутствует, программа не должна молча считать локальную
зону UTC: создаётся предупреждение и требуется политика пользователя.

Реализованный ISO-8601 normalizer принимает строки с `Z` или явным UTC offset, переводит их
в `datetime64[ns]` UTC и сохраняет исходный offset. Naive ISO-строки остаются в исходном
wall-clock представлении с `timezone=None` и предупреждением. Смесь aware/naive значений не
считается надёжным DATETIME-кандидатом. Пустые элементы становятся `NaT` с диагностикой.

Реализована также явная нормализация пользовательского формата и пары DATE+TIME. Форматы
задаются синтаксисом `datetime.strptime`, колонки обязаны иметь одинаковую длину, а неполная
пара становится `NaT`. Пользователь может выбрать IANA zone (`Asia/Oral`) или фиксированный
offset (`UTC+05:00`). Для DST-перехода неоднозначное/несуществующее локальное время не
угадывается: операция требует строки с явным offset.

Автоопределение Unix-разрядности проверяет правдоподобный диапазон дат и не основывается
только на количестве цифр.

## 6. TIME ↔ DEPTH

Преобразование не является взаимно однозначным. Одна глубина может встречаться при
бурении, остановке, подъёме и повторном прохождении.

`TimeDepthMappingService` создаёт новый derived dataset или mapping profile и требует:

- исходный временной индекс;
- целевой глубинный индекс;
- правило повторов: first/last/min/max/mean;
- шаг или интервалы агрегации;
- политику неподвижной глубины и обратного движения.

В ядре уже реализован nearest TIME→DEPTH resolver. Строгая `error`-политика
остаётся по умолчанию, а `first`, `last`, `min`, `max` и `mean` требуют явного
выбора. Результат возвращает все совпавшие строки; для среднего значения
исходная строка не подменяется и указывается как `None`.

Выбор dataset, TIME/DEPTH индексов и политики сохраняется в versioned project
mapping-профиле. При загрузке ссылки на dataset и роли индексов валидируются.
Расширенный UI preview интервов остаётся следующим этапом.

Ядро интервальной агрегации реализовано как неразрушающая операция: relative TIME
и DATETIME разбиваются на окна заданного шага, глубина следует политике
профиля, а кривые усредняются только по конечным значениям. Создаётся новый
derived dataset с TIME/DEPTH индексами и provenance; исходник не изменяется. Preview-план
проверяет отпечаток индексов и кривых перед применением. UI preview остаётся следующим
инкрементом.

Исходный временной dataset сохраняется. Результат получает provenance с параметрами
сопоставления. Скрытое удаление дублей глубины запрещено.

## 7. Контроль качества

`DataQualityService` выдаёт структурированные `DataIssue` с кодом, severity, колонкой,
диапазоном строк и рекомендацией. Минимальный набор:

- отсутствие или неоднозначность индекса;
- нераспознанное время/часовой пояс;
- обратный порядок, дубли и разрывы;
- неравномерный шаг;
- несовпадение STRT/STOP/STEP с данными;
- полностью пустые/постоянные кривые;
- несовместимые единицы и выбросы.

Ошибки блокируют импорт/экспорт только по явной политике. Предупреждения видны до
редактирования и сохраняются в проекте.

## 8. Редактирование и версии данных

Индексные колонки редактируются тем же command/history-механизмом, но команды получают
признак повышенного риска. Перед изменением времени/глубины UI показывает влияние на
сортировку, mapping и derived curves.

Автоматическая сортировка, ресэмплинг, интерполяция, агрегация и объединение создают
derived dataset либо подтверждённую команду. Source fingerprint, исходный файл и журнал
операций позволяют сравнить исходную и текущую версии без обязательного дублирования
всех больших массивов в памяти.

## 9. Визуализация

- Depth mode: общая вертикальная инвертированная ось глубины и независимые X-треки.
- Time mode: ось времени с UTC/локальным представлением, пропусками и событиями.
- Один dataset может переключать active index без удаления остальных индексов.
- Для TIME+DEPTH одна кривая может иметь два представления, но оба используют общие
  исходные значения и явный mapping.

Текущий `TabletView` развивается как depth renderer. Time renderer использует общую
декларативную модель стилей и selection, но не должен имитировать время через поле depth.

## 10. Экспорт

`ExportPlan` до записи показывает выбранный формат, индекс, кривые, NULL, единицы,
версию LAS и ожидаемые потери. Если LAS-версия не представляет исходное время, UI
предлагает timestamp, DATE+TIME либо CSV/Parquet/JSON.

Экспорт обязан сохранять порядок кривых, описания, единицы, метаданные и provenance.
Необратимая потеря второй индексной колонки без предупреждения запрещена.

## 12. Этапы внедрения без поломки проекта

1. [готово] Чистые модели `DatasetIndex`, `IndexCandidate` и тестовые фикстуры.
2. [готово] `IndexDetectionService` для числовых DEPTH/TIME, datetime64 и Unix timestamp.
3. [готово для сохранённых индексов] Data Inspector с confidence, evidence и ручным выбором;
   review всех сырых колонок будет расширен в Universal Import.
4. [готово] Несколько индексов появились в v6 и сохраняются текущим форматом проекта v16.
5. Временной normalization/QC и строковые даты.
6. Strict/compatible/manual import pipeline.
7. Time renderer и выбор active index.
8. [частично] Безопасный nearest-row TIME↔DEPTH resolver для событий: ровно одна пара индексов,
   отказ при неоднозначной глубине; derived mappings, повторы и профили агрегации запланированы.
9. Экспорт с анализом потерь и временная печать.

## 13. LAS Editor и композиция файлов

Редактор работает с LAS 1.2/2.0 и совместимо распознанными новыми/нестандартными LAS
через одну нормализованную модель. Он предоставляет связанную таблицу и график: выбор
строк или диапазона в одном представлении подсвечивает тот же диапазон в другом.

Новый LAS создаётся с явно выбранными версией, индексом, единицей, START/STOP/STEP, NULL
и каталогом кривых. Шаг является любым конечным положительным числом (например 0.1, 0.2,
0.5 или 1 м), а не фиксированной настройкой приложения.

Сращивание datasets и вставка GIS/GTI-кривых выполняются через `DatasetMergePlan`:

- выбор главного индекса и целевой сетки;
- проверка единиц и диапазонов;
- стратегия совпадающих мнемоник;
- интерполяция/агрегация только после подтверждения;
- preview добавленных, заменённых и пропущенных значений;
- новый dataset с provenance всех источников.

Убывающая глубина не считается повреждением исходника. `DataQualityService` сообщает
направление, а команда исправления создаёт derived-копию, синхронно разворачивая индекс и
все кривые. Mixed depth нельзя исправлять простым reverse — требуется сортировка/правило
повторных проходов и отдельный preview.

## 14. Единый таблично-графический редактор

Таблица и график используют один `SelectionModel`: интервал индекса, выбранные curve IDs и
режим выбора. Команды не зависят от места запуска — действие из контекстного меню графика
и действие над выделенными ячейками создают одинаковую command/history-запись.

Диапазонные операции включают постоянное значение, случайный диапазон, очистку, формулу,
копирование и вставку. Случайная операция хранит seed, поэтому redo и аудит полностью
воспроизводимы. Многоканальное изменение атомарно: либо применяются все кривые, либо ни
одна.

После изменения исходных газовых компонентов немедленно пересчитываются Total Gas и
доступные отношения. Расчётные каналы по умолчанию read-only: пользователь изменяет их
исходные компоненты либо создаёт независимую копию кривой.

Та же selection передаётся в экспорт. TXT/CSV и Excel содержат только выбранные строки и
параметры; Excel разделяет числовые данные и метаданные по листам. Экспорт не изменяет
dataset и выполняется атомарно через временный файл.

## 12. Ежедневное наращивание нескольких datasets (0.7.45)

Скважина может содержать любое количество независимых DEPTH и TIME datasets. Пользователь
явно выбирает приёмник; совпадения только по слову DEPTH/TIME недостаточно. До append проверяются
role/type/unit индекса, WELL, полный набор мнемоник и единицы кривых. Идентичное перекрытие
пропускается, конфликтующее блокирует операцию, а новые строки обязаны образовывать монотонный
суффикс. SHA-256 уже импортированного файла превращает повторную операцию в no-op.

Изменяется только выбранный dataset. Остальные глубинки, временки, derived datasets, формы,
комментарии и значки не затрагиваются. После успешного append создаётся `DatasetAppendRecord`;
project format v20 хранит отдельный `append_history` у каждого dataset. Исходные суточные LAS
остаются неизменными первичными файлами, а проект становится проверяемой составной рабочей копией.
