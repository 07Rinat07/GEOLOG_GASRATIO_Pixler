# ООП-архитектура взаимодействия планшета

## Назначение

Архитектура отделяет отрисовку от обработки ввода и не позволяет одному прозрачному виджету заблокировать редактирование всего планшета.

## Компоненты

### `TabletInteractionRouter`

Получает нормализованные `TabletInputEvent` и последовательно передаёт их зарегистрированным обработчикам. Только маршрутизатор хранит логическое состояние активного жеста. Нативный `grabMouse()` не используется.

Приоритет обработчиков:

1. существующая аннотация;
2. активный инструмент создания аннотации;
3. выбор/редактирование дорожки;
4. штатная логика кривой, геологических интервалов и навигации.

### `TabletEditModeCoordinator`

Централизует инварианты F4:

- аннотации и дорожки включаются одновременно;
- дорожка временно приостанавливается только при выбранном инструменте создания;
- отключение F4 обязательно снимает инструмент и восстанавливает курсоры;
- состояние нельзя частично изменить напрямую из виджета.

### `AnnotationInteractionHandler`

Не рисует интерфейс. Управляет сценарием выбора, создания, drag/resize, двойного щелчка, клавиш F2/Enter/Delete и контекстного меню. Изменение геометрии отправляется в модель один раз после отпускания мыши.

### `TrackEditInteractionHandler`

Одинарный щелчок выбирает дорожку, но пропускает событие дальше для выбора кривой. Двойной щелчок открывает полный редактор подходящей дорожки.

### `TabletAnnotationOverlay`

Только рисует и выполняет hit-test. Виджет постоянно имеет `WA_TransparentForMouseEvents`, не содержит обработчиков мыши, не устанавливает mask и не захватывает указатель. При drag/resize обновляется только dirty-область объекта.

### `TabletInteractionWatchdog`

Проверяет реальное состояние левой кнопки во время логического жеста. Если Windows не доставила release после Alt+Tab, модального окна или перехода между мониторами, watchdog завершает жест и освобождает маршрутизатор.

## Правила расширения

Новый инструмент планшета реализует интерфейс обработчика маршрутизатора. Он не должен устанавливать собственный event filter на все дорожки, вызывать `grabMouse()` или управлять флагами другого инструмента. Переключение режимов выполняется через координатор.

## Проверки инвариантов

- paint-overlay не содержит mouse event handlers;
- в коде overlay отсутствуют `grabMouse`, `releaseMouse` и `setMask`;
- один drag создаёт одно изменение геометрии;
- щелчок по пустому графику в F4 выбирает дорожку и остаётся доступным кривой;
- существующая аннотация имеет приоритет над созданием новой;
- после снятия инструмента двойной щелчок снова открывает редактор дорожки;
- деактивация окна и потеря release очищают активное состояние.

## Граница изменения layout — 0.7.31

`TabletView` отвечает за Qt-жесты, локальную геометрию и перерисовку. Изменение
сериализуемого `TabletLayout` выполняется через `TabletLayoutMutationController` либо через
синхронный request в `TabletController` основного окна. Это относится к ширине и порядку
треков, вертикальному индексу и видимому диапазону.

Для resize/reorder request отправляется до локального fallback. Если основное приложение
обработало сигнал и изменило общий layout, повторной мутации нет. Standalone-тесты и отдельное
использование view сохраняют работоспособность через headless fallback. Undo/Redo повторяет тот
же путь, поэтому состояние проекта не обходится прямым присваиванием из Qt-класса.

Source-integrity тест запрещает возврат прямых вызовов `TabletLayout.set_*`, `move_track`,
`add_track`, `remove_track` и присваивания `TrackDefinition.width` в `TabletView`.
