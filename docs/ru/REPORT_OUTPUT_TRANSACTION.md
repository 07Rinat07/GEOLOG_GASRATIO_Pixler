# Восстанавливаемая транзакция отчётных файлов

Статус: реализовано в версии 0.7.39. Runtime schema журнала: v1. Report Passport: schema v4.
Формат проекта остаётся v16.

## Назначение

Файловый отчёт и его `*.passport.json` теперь фиксируются как одна восстанавливаемая операция.
Это не обещает недоступную большинству файловых систем мгновенную атомарность нескольких файлов,
но гарантирует журналируемый commit, полный rollback и повторное восстановление после прерывания
процесса или сбоя драйвера/диска.

Покрытые сценарии:

- Print Center PDF и постраничные PNG/JPEG/TIFF/BMP/WebP/SVG;
- прямой PNG/SVG/PDF активной визуализации;
- CSV/XLSX выбранного интервала;
- Masterlog PDF;
- интерпретационный PDF.

## Последовательность

1. При наличии старого журнала выполняется recovery.
2. Renderer/exporter пишет только в скрытый staging-каталог рядом с целевым файлом.
3. Проверяются непустые output-файлы и безопасные относительные имена.
4. По фактически записанным байтам рассчитываются SHA-256, размер и MIME-type.
5. Report Passport schema v4 получает список `artifacts` и заново подписывается.
6. Sidecar формируется в staging-каталоге.
7. Журнал фиксирует операции и резервные копии существующих файлов.
8. Старые output/sidecar перемещаются в backup, новые файлы устанавливаются через `os.replace`.
9. Установленные файлы повторно проверяются по passport fingerprints.
10. Только после этого журнал получает состояние `committed`; backup, staging и журнал удаляются.

## Recovery

До `committed` незавершённая транзакция откатывается:

- новые частично установленные файлы удаляются;
- прежние output и sidecar восстанавливаются из backup;
- устаревшие страницы продолжения возвращаются при rollback;
- staging и журнал удаляются только после успешного восстановления.

Если commit уже подтверждён, но процесс завершился до очистки, recovery сохраняет новую пару и
удаляет только backup/staging/journal.

Следующий экспорт в тот же путь запускает recovery автоматически. Для ручной проверки каталога:

```powershell
.\.venv\Scripts\python.exe tools\recover_report_transactions.py "C:\Reports"
```

## Output fingerprint

Report Passport schema v4 содержит для каждого готового файла:

- только безопасное имя файла без абсолютного пути;
- роль `single-file` или `page`;
- номер страницы для постраничного результата;
- MIME-type;
- размер в байтах;
- SHA-256 фактического содержимого.

`load_report_passport()` проверяет как SHA-256 JSON, так и наличие, размер и digest каждого
output artifact. Изменение уже сформированного PDF, изображения, CSV или XLSX обнаруживается.

## Перезапись и страницы продолжения

При подтверждённой перезаписи транзакция резервирует прежнюю пару. Для постраничного экспорта
она также удаляет лишние старые `*_page_NNN.*`, если новый отчёт содержит меньше страниц.
Удаление входит в тот же rollback-контракт.

## Ограничения

- journal содержит абсолютные служебные пути, но они никогда не входят в Report Passport;
- recovery разрешает операции только внутри каталога output и собственного staging-каталога;
- физическая печать не создаёт файл и поэтому не имеет output fingerprint;
- электронная подпись организации и доверенный timestamp этим механизмом не заменяются;
- аппаратные и сетевые файловые системы всё равно требуют Windows smoke-test.

## DOCX и HTML в 0.7.40

DOCX и HTML формируются только в staging и устанавливаются вместе с Passport v4. Транзакция
проверяет непустой готовый файл, рассчитывает MIME, размер и SHA-256, а при сбое восстанавливает
предыдущий document и sidecar.
